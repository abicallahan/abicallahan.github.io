Birthday Paradox

The birthday paradox says that if the number of people in a room is more than 23, the probability that two people in the room share a birthday is more than 50%. I wrote a program in Java to simulate this paradox.

In this project, the program starts by creating one experiment for 5 people and then runs 20 trials with 5 people. Each trial has 5 different people with randomly generated birthdays. The program checks if any two people in the trial share a birthday, and if so, it counts that trial as a success. After 20 trials, the program calculates the percentage of trials that were successful and prints that percentage to the user. The program then repeats this process for 10 people, 15 people, and so on until it reaches 100 people. The results show that when there are 23 people in the room, the percentage of trials that were successful is around 50%, which confirms the birthday paradox.

In this project, I worked with a partner to write the program. I was responsible for writing the code for the Birthday Class as well as the Experiment Class. After working through the code with my partner, we both went through the remainder of the project, collaborating on the code and debugging together.

This project was interesting to me because I had heard of the birthday paradox before, but I had never seen a program that simulates it. I thought it was really cool to see the results of the program and how it confirmed the paradox. I also enjoyed working with a partner on this project and collaborating on the code together. Since it was the first project of the semester, it was a great refresher on Java and how to write a program in Java. Elevator Simulator

In this project, I worked with my partner to develop an elevator simulator that services several different types of elevators and schedulers for the elevators in a building with a given number of floors. The four different elevators have different behaviors that they follow, and each scheduler works differently in assigning calls to elevators. Schedulers will take in Calls that give a starting and ending floor.

In this project, there are four different types of elevators: standard, freight, express, and penthouse. The standard elevator visits all of the floors, and will accept a new call if the pick up and drop off floors of the call are in the same direction that it's already traveling and would not cause it to change direction. It will also accept a new call if it is waiting on the first floor. The freight elevator can visit all of the floors, but it will not change direction to pick up a call, it only holds one person at a time, and it will change direction to answer a call if it is headed to the ground floor. The express elevator only services the top half of the building and the ground floor. The penthouse elevator only travels between the top and ground floor. The two schedulers that exist are the simple scheduelr and the complex scheduler. The simple scheduler assigns calls from left to right, and the complex scheduler assigns calls based on the elevator that can complete the call in the least amount of rounds.

Specifically, this project is best implemented through the use of Abstract classes and interfaces which allow us to group common functionality/attributes for a robust and abstracted design. Both our Elevator and Scheduler classes were abstract so that they shared methods such as moveUp, moveDown, and assignCall. Our design ensures that we follow the SRP as each of our classes is doing only one thing, and having only one reason to change. The code supports open closed principle as our classes are programmed to abstractions (method taking in a type Elevator rather than creating it.), Additionally, since we took time to properly design parent/subclass structure, substituting parent class objects for subclass objects did not break our program. For dependency inversion, we ensured that our high level scheduler relies on low level elevators. Finally, our interfaces were created narrowly enough so that implementing classes did not have to implement unnecessary methods.

In this project, I worked with a partner to write the program. I was responsible for writing the code for two of the Elevators as well as the Simple Scheduler. I also worked to implement the Call Class and Direction Enum. After working through the code with my partner, we both went through the remainder of the project, collaborating on the code, refactoring, and debugging together. Each class was accompanied by a test file that was created through TDD.

This project was interesting to me because I had never gone through Test Driven Development before. It was a new experience for me that I now have a lot of knowledge in. Although at some times, TDD can be tedious, it works in the long run especially getting a feel for how our code is supposed to work.
